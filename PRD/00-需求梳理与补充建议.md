## 2. 一句话结论

这是一个面向企业内部消防管理的微信小程序 + Web 管理后台项目，标准方案核心范围是：组织/人员权限准入、消防报修工单、消防培训答题、培训证书、消息通知、数据看板、数据导出和操作日志。

当前文档在商务范围、报价、验收、上线配合方面已经比较完整；但在开发落地前，还需要补齐权限矩阵、账号绑定规则、页面原型、数据字典、接口/第三方服务清单、Excel 模板、消息模板、证书模板、运维与安全细则。

## 3. 当前标准方案范围

### 3.1 产品形态

| 端 | 说明 |
| --- | --- |
| 微信小程序 | 员工和企业管理员使用，包含登录、权限申请、报修、培训、证书、消息、个人中心 |
| Web 管理后台 | 平台侧使用，包含企业管理、用户管理、权限审批、报修处理、培训管理、看板、导出、日志 |

### 3.2 组织与角色

组织关系默认为：总部 → 子公司 → 员工。

| 角色 | 使用端 | 主要职责 |
| --- | --- | --- |
| 超级管理员 | 管理后台 | 系统配置、全部数据管理 |
| 平台运营 | 管理后台 | 审批权限、处理报修、发布培训、查看数据 |
| 企业管理员 | 小程序为主，是否进入后台需确认 | 申请员工/子公司权限、代提交报修、查看企业范围数据 |
| 员工 | 微信小程序 | 提交报修、参加培训、查看个人记录 |

需要注意：部分文档写到“管理后台角色：超级管理员、平台运营、企业管理员”，但详细说明又写“企业管理员使用小程序”。开发前应统一企业管理员是否拥有 Web 后台入口。

| 模块 | 本次是否包含 | 主要内容 |
| --- | --- | --- |

| 组织管理 | 包含 | 总部/子公司/员工关系，企业启停，数据隔离 |
| 权限开通工单 | 包含 | 企业管理员发起，后台审批通过/驳回，结果通知 |

## 4. 主流程梳理

### 4.1 权限准入流程

1. 企业管理员发起开通申请。
2. 申请对象为子公司或员工。
3. 平台运营在后台审批。
4. 审批通过后账号获得对应权限。
5. 审批驳回时填写原因，申请人可查看。
6. 审批结果通过站内消息和微信订阅消息通知。

待明确点：第一个企业管理员如何产生、员工微信身份如何与手机号/姓名绑定、员工是否能查看自己的申请状态。

### 4.2 消防报修流程

1. 员工或有权限的企业管理员在小程序提交报修。
2. 报修字段包括紧急程度、故障类型、位置、描述、联系人、电话、现场照片。
3. 系统生成唯一工单号，初始状态为待受理。
4. 后台平台运营受理，状态进入处理中。
5. 后台可派单、添加处理备注、更新进度。
6. 工单最终标记已完成或已关闭。
7. 状态变更通知提交人。
8. 后台支持报修数据导出。

待明确点：已完成和已关闭的区别、是否存在驳回/撤销/重新打开、派单对象是谁、处理人是否登录系统。

### 4.3 消防培训流程

1. 后台维护题库，支持单选、多选、判断。
2. 支持 Excel 模板批量导入题库。
3. 后台创建培训任务，设置起止时间、及格分、最大作答次数、参训对象和关联题目。
4. 员工在小程序查看任务并答题。
5. 系统判分，默认 60 分及格，最多 3 次，取最高分。
6. 后台留存成绩和答题明细。
7. 通过后生成培训证书。
8. 后台支持培训记录导出。

待明确点：是否随机抽题、是否显示答案解析、多选题计分规则、任务过期后的补考机制、证书模板。

### 4.4 上线配合流程

1. 甲方准备营业执照、法人/管理员身份证、小程序名称/简介/Logo、隐私政策、培训题库。
2. 甲方购买域名和云服务器，并完成域名实名。
3. 甲方完成 ICP 备案。
4. 甲方注册并认证微信小程序。
5. 甲方向承建方提供 AppID、隐私协议页面地址等信息。
6. 承建方开发、部署、提审小程序。
7. 微信审核通过后发布。

建议立项后立即启动备案和小程序认证，因为备案 7 ~ 20 个工作日，小程序审核通常还需预留 1 ~ 2 轮修改。

## 8. 当前文档里的风险点

1. 企业管理员使用端不一致：有的文档写管理后台包含企业管理员角色，有的文档写企业管理员只使用小程序。
2. 权限开通存在起点问题：如果员工不能自助申请，首个企业管理员和员工批量初始化流程必须明确。
3. 子公司创建方式有重叠：后台企业管理可新增子公司，同时权限开通工单也可申请开通子公司，需要明确主流程。
4. 报修状态不完全一致：主文档为待受理、处理中、已完成、已关闭；详细文档又出现已驳回。
5. 派单对象不清楚：如果没有维保单位角色，派单给谁、是否需要处理人账号、处理人能否登录系统需要确认。
6. 微信订阅消息不能保证必达：用户授权、模板限制、发送次数都会影响通知效果，需要站内消息兜底。
7. 证书功能容易被低估：PDF/图片生成、保存相册、模板、编号、防伪、历史证书查看都需要细化。
8. 验收用例覆盖偏粗：缺少异常流、边界值、权限矩阵、导入失败、图片上传失败、网络异常等测试。
9. 安全合规只到确认层：缺少密码策略、后台登录保护、数据备份、敏感信息脱敏、日志审计等落地细则。
10. 非功能指标较少：仅有 4G 首屏 3 秒、50 人同时在线，缺少可用性、备份恢复、图片容量、数据库容量等指标。

## 10. 建议优先向甲方确认的问题

### 10.1 权限与账号

1. 首个总部企业管理员由平台后台创建，还是通过工单申请？
2. 员工开通后，系统如何匹配其微信账号：手机号、邀请码、管理员绑定，还是后台预置？
3. 企业管理员是否需要 Web 管理后台权限，还是只在小程序操作？
4. 总部企业管理员是否可查看/管理所有子公司员工和培训记录？
5. 员工离职、调岗、换手机号、换微信时如何处理？

### 10.2 报修工单

1. 工单是否允许提交人撤销？
2. 工单是否允许平台驳回？若允许，状态如何展示？
3. 已完成是否需要提交人确认后再关闭？
4. 派单对象是平台内部人员、企业人员，还是仅作为备注字段？
5. 是否需要记录每个处理节点的照片或附件？

### 10.3 培训考试

1. 题目是固定试卷，还是按题库随机抽题？
2. 多选题少选、错选如何计分？
3. 是否展示正确答案和解析？
4. 员工 3 次都未通过后如何处理？
5. 证书是否需要二维码验真或证书编号查询？

### 10.4 产品体验与上线

1. 小程序名称、Logo、品牌主色、服务类目是否已确定？
2. 是否已有隐私政策/用户协议，还是需要承建方提供模板？
3. 云服务商选阿里云还是腾讯云，账号由谁持有？
4. 是否已有域名和备案主体？
5. 小程序审核测试账号由谁提供？

## 11. 推荐执行顺序

1. 先让甲方确认 B-01 ~ B-10、P-01 ~ P-05、C-01 ~ C-05，冻结标准方案或增减项。
2. 同步启动域名、云服务器、ICP 备案、小程序注册认证。
3. 补齐权限矩阵、页面清单、字段清单、报修状态机、培训判分规则。
4. 确认题库 Excel 模板、消息模板、证书模板。
5. 形成开发版 PRD 和验收版测试计划。
6. 再进入 UI、接口、数据库、开发排期和实施。

## 12. 建议范围策略

建议本期按“标准方案”推进，不建议首期加入巡检、设备台账、视频培训、HR/OA 对接、外部维保系统对接等扩展能力。原因是当前标准方案已经包含组织权限、工单、培训、证书、通知、看板和导出，业务闭环完整，适合 2 ~ 3 个月一次性交付。

如果甲方强需求更完整的消防数字化体系，建议优先规划以下二期能力：

| 优先级 | 二期方向 | 原因 |
| --- | --- | --- |
| 高 | 消防设备台账 | 报修和巡检都需要设备基础数据支撑 |
| 高 | 消防巡检管理 | 消防管理常态化需求，比单纯报修更有业务价值 |
| 中高 | 视频课程培训 | 若存在学时合规要求，纯答题可能不够 |
| 中 | HR/OA 对接 | 员工数量大时可减少人工开通维护成本 |
| 中 | 隐患随手拍 | 可覆盖消防通道堵塞、安全出口锁闭等非设备故障场景 |
