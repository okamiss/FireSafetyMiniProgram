# AGENTS.md Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create a root-level `AGENTS.md` that gives coding agents a complete, repository-grounded collaboration guide for architecture, engineering conventions, testing, and business boundaries.

**Architecture:** Use one root instruction file whose scope covers `server`, `admin-web`, `miniapp`, `deploy`, and project documentation. The file will distinguish verified current behavior from future implementation rules and will point agents to manifests and business documents as the source of truth.

**Tech Stack:** Markdown, Spring Boot 3.5.3 / Java 21 / Maven, Vue 3 / TypeScript / Vite, Element Plus, uni-app, MySQL 8.4, Docker Compose, Nginx.

- [ ] **Step 1: Write the root guide**

Create `AGENTS.md` with all of the following explicit sections and rules:

1. Scope and instruction priority: the file applies to the whole repository; direct user instructions win; current code and manifests win over stale prose.
2. Project status: this is a three-part framework under active development, not a claim that every documented business module is implemented.
3. Repository architecture: define responsibilities for `server`, `admin-web`, `miniapp`, `deploy`, `docs`, and root business documents.
4. Runtime flow: both clients call `/api`; admin Vite proxies to port 8080; miniapp currently contains a placeholder API origin; MySQL is persisted by Compose; Redis is provisioned but not yet used by a declared backend dependency.
5. Actual stack and versions from the repository manifests.
6. Change workflow: inspect status and nearby code first, make the smallest compatible change, preserve unrelated work, and update docs when behavior or commands change.
7. Context7 rule: for any library, framework, SDK, API, CLI, or cloud-service question, call `resolve-library-id` first unless an exact `/org/project` ID is supplied, choose the best relevant match, and call `query-docs` with the full question before answering or implementing.
8. Source handling: edit `.ts` and `.vue` sources; do not directly edit tracked emitted `.js` files or `admin-web/tsconfig.tsbuildinfo`; do not commit `node_modules`, `dist`, `target`, logs, or secrets.
9. Backend conventions: package by feature, keep `/api` routes and `ApiResponse<T>`, use DTO validation, enforce authentication/authorization and enterprise isolation server-side, keep state transitions explicit, add tests, and never bypass `ddl-auto: validate` by silently enabling destructive automatic schema creation.
10. Admin conventions: strict TypeScript, Vue SFCs, shared Axios client, router metadata, Element Plus, Chinese user-facing copy, and no duplicated API/auth logic.
11. Miniapp conventions: use uni-app APIs and storage, centralize requests/session handling, configure the API origin per environment before integration, and account for WeChat permissions and lifecycle behavior.
12. Cross-cutting rules: consistent API contracts, Asia/Shanghai-aware business time handling, safe uploads/exports, audit logging for important state changes, and no sensitive data in fixtures.
13. Business scope: organization/users, permission work orders, repair work orders, training/tasks/records, certificates, site/WeChat messages, dashboard, Excel import/export, and operation logs.
14. Business invariants: enterprise data isolation, RBAC, auditable approvals, explicit repair and permission state machines, reproducible training scores/certificates, and site-message fallback for WeChat notifications.
15. Scope exclusions: do not add patrol, equipment ledger, hidden-danger reporting, HR/OA, IoT, enterprise WeChat, or other extension-analysis items without explicit approval.
16. External dependencies: do not promise guaranteed subscription-message delivery, WeChat approval/filing dates, third-party interface behavior, or an absolute network-performance target.
17. Verification commands: backend Maven tests, admin `npm ci` and build, miniapp `npm ci` and WeChat build, Docker Compose config validation, and the `/api/health` smoke check.
18. Definition of done: relevant verification passes, failures are reported accurately, docs are synchronized, no unrelated files or secrets are included, and business-scope changes receive explicit confirmation.
