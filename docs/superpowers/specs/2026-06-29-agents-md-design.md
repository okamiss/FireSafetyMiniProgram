# AGENTS.md 协作指南设计

## 目标

在仓库根目录创建一份完整的 `AGENTS.md`，为后续编码代理提供统一、可执行的项目协作规则。指南必须以当前仓库的实际结构、依赖、脚本和业务文档为准，避免套用与项目不符的通用模板。

## 文件范围

采用单一根目录文件，作用域覆盖整个仓库：

- `server`：Spring Boot REST API、领域逻辑、持久化和后端测试。
- `admin-web`：Vue 3 管理后台。
- `miniapp`：uni-app 微信小程序。
- `deploy`：Docker Compose、Nginx 和部署配置。
- `docs` 及根目录业务文档：需求、验收和交付依据。

当前项目规模不需要为各子目录增加独立 `AGENTS.md`。如后续模块形成明显不同的开发与验证规则，再拆分子级文件。

## 内容结构

最终指南应包含以下部分：

1. 项目定位、一次性交付原则和标准范围。
2. 仓库架构、各目录职责以及跨端协作关系。
3. 当前技术栈、关键版本和配置来源。
4. 修改前检查、最小范围变更和文档优先级。
5. Java、Vue、TypeScript、uni-app 的项目内编码约定。
6. API 响应、认证、权限隔离、状态机和数据安全约束。
7. 权限工单、消防报修、培训、证书、消息、看板、导出和日志的业务边界。
8. 微信平台审核、订阅消息送达、第三方接口和性能指标等外部依赖边界。
9. 后端测试、后台构建、小程序构建和 Docker 配置验证命令。
10. 完成定义、文档同步要求和禁止事项。
11. Context7 查询规则：涉及框架、库、SDK、API、CLI 或云服务时，先解析库 ID，再按完整问题查询当前文档。

## 关键约束

- 所有规则必须与仓库中真实存在的脚本和配置一致。
- 优先修改 `.ts` 和 `.vue` 源文件；`admin-web` 中已跟踪的同名 `.js` 与 `tsconfig.tsbuildinfo` 不作为手工修改入口。
- 不擅自加入巡检、设备台账、HR/OA、IoT 等未确认扩展范围。
- 不把微信订阅消息、平台审核、备案、第三方接口或网络性能写成无条件必达承诺。
- 不提交真实密钥、令牌、生产密码或包含个人信息的测试数据。
- 后端保持统一响应结构和 `/api` 路径；新增领域功能需要有相应测试。
- 变更应保持小范围、兼容现有结构，并在交付前运行与改动对应的验证命令。

## 验证设计

生成 `AGENTS.md` 后执行以下检查：

- 检查文件包含架构、规范、测试、业务边界和 Context7 规则。
- 对照 `README.md`、`pom.xml`、两个 `package.json`、`docker-compose.yml` 和现有源码，核对版本、路径、端口与命令。
- 确认没有占位符、矛盾规则或未在仓库中存在的脚本。
- 使用 `git diff --check` 检查格式问题，并查看最终差异确保没有改动无关文件。

## 非目标

- 本任务不修改业务代码、依赖、部署配置或现有需求文档。
- 本任务不重新设计系统架构，也不扩充一次性交付范围。
- 本任务不处理仓库中已有的编译产物；仅在指南中说明后续代理应如何对待它们。
